## TAREFA 1 — Kelly Fractional Sizing

### Implementação

**`scripts/backtest.py`:**
- `BacktestConfig.enable_kelly: bool = False`
- `_kelly_size_factor(trades, n=50, fraction=0.25, max_pos_pct)`:
  - W = win rate dos últimos 50 trades
  - R = avg_win_eur / avg_loss_eur
  - `f* = W - (1-W)/R` (full Kelly)
  - `f_quarter = 0.25 × max(0, f*)` — Quarter Kelly
  - `scale = min(1.0, f_quarter / (max_pos_pct/100))` — relativo ao cap
  - Floor: `0.5` se `f* ≤ 0` (edge negativo)
  - Não aplica nos primeiros 10 trades (dados insuficientes → scale=1.0)
- CLI: `--kelly` corre comparação Kelly-ON vs Kelly-OFF (Full variant)

**`bot/cro.py`:**
- `_kelly_size_factor(closed, n=50, fraction=0.25, max_pos_pct)` — versão para dicts beta_trades
- `_validate_proposal()` aplica quando `CRO_CONFIG["enable_kelly_sizing"] = True`

### Diagnóstico — Por que Kelly falhou neste sistema

Com WR=37.6% e R≈2.4 (avg_win/avg_loss):
- `f* = 0.376 - (1-0.376)/2.4 = 0.376 - 0.260 = 0.116` (11.6% de equity)
- Quarter-Kelly: `0.25 × 0.116 = 2.9%` de equity
- Scale vs max_pos_pct=11%: `2.9/11 = 0.27` → cada posição reduzida para **27% do tamanho máximo**

O resultado: capital deployado cai drasticamente, trades mais pequenos não compensam o número de perdas, retorno colapsa de +224.5% → +65.6%.

**Quarter-Kelly é inadequado para sistemas com WR < 40%.** O sistema v3 tem um perfil assimétrico (baixo WR, alto R:R) onde o Kelly ideal seria de tamanho completo ou meio-Kelly, mas nunca quarter. Para o v3, a gestão de risco via `max_position_pct` (cap duro) é mais adequada.

### Veredicto: Kelly-OFF mantido. CRO_CONFIG["enable_kelly_sizing"] = False (default).

### Problema identificado: Label Mismatch

Bonnie v3 é **quase pass-through** (filtra < 0.3% dos sinais vs ~30% da v2). A causa é:

- **Labels de treino**: `+1.5×ATR TP / -1.0×ATR SL` em 20 dias
- **Parâmetros reais v3**: `+4.25×ATR TP / -1.75×ATR SL`

O modelo foi treinado para prever um alvo completamente diferente do que a estratégia agora executa. Com label balance ~45% e accuracy CV ~50.6% (quase aleatório), o threshold optimizer encontra `0.30` como ótimo porque maximiza recall (passa tudo) em detrimento da precisão.

## Lições e Próximos Passos

1. **Kelly OFF é o correto para este sistema.** Com WR ~38% e R:R assimétrico, Quarter-Kelly reduz posições para ~27% do tamanho máximo → colapso de retorno sem compensação de risco suficiente. O controlo de risco via `max_position_pct=11%` é mais adequado. Kelly só faz sentido se WR > 50%.

2. **Bonnie v3 requer recalibração de labels.** Para o modelo ser útil com os parâmetros v3 (atr_tp=4.25, atr_stop=1.75), o corpus deve ser gerado com `TP_ATR_MULT=4.25` e `SL_ATR_MULT=1.75`. Este é o único caminho para uma Bonnie v4 funcional.

3. **Bonnie v2 confirmada como modelo ativo.** `bonnie_model_v3.pkl` apagado (2026-05-24). BonnieML carrega v2 automaticamente. Backtest de confirmação (2024-01-01→2026-05-01) validou: retorno **+39.6%**, Bonnie filtra **27.3%** dos sinais, setup log confirma `v2 loaded`.

4. **7yr full run confirma v3 params:** +224.5% vs SPY +232.3% = alpha -7.8pp. Mais cauteloso que o Learner reportou (+231.2%) devido a variação natural de dados entre runs.
